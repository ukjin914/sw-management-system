# AI 통합 검색 기능 도입 계획

## 1. 개요

현재 SW Management System의 검색 기능(10개 도메인, 30개+ 검색기능)을
자연어 기반 AI 통합 검색으로 확장하는 계획

### 목표
- 사용자가 자연어로 질문하면 시스템이 알아서 검색하여 결과 반환
- 모든 데이터 처리는 로컬 서버에서 수행 (외부 유출 없음)
- 기존 검색 기능은 그대로 유지, AI 검색은 추가 기능으로 제공

### 예시
```
입력: "서울 지역 진행 중인 계약 알려줘"
결과: contract 테이블에서 city_nm='서울', progress_status='IN_PROGRESS' 조회 후 결과 반환

입력: "김과장이 담당한 이번 달 업무계획"
결과: work_plan + users 조인하여 해당 조건 조회 후 결과 반환

입력: "이번 달 만료되는 라이선스"
결과: license_registry에서 만료일 기준 조회 후 결과 반환
```

---

## 2. 아키텍처

```
[브라우저] → 자연어 질문 입력
    ↓
[Spring Boot] POST /api/ai/search
    ↓
[Ollama (localhost:11434)] 자연어 → JSON 검색조건 변환
    ↓  예: {"domain":"contract","filters":{"city":"서울","status":"IN_PROGRESS"}}
[Spring Boot] 기존 Repository로 DB 조회
    ↓
[브라우저] ← 결과를 테이블/카드 형태로 응답
```

### 핵심 포인트
- Ollama는 로컬에서 실행되는 LLM 서버 (무료, 오픈소스)
- AI는 검색 조건 파싱만 담당, 실제 데이터 조회는 기존 JPA Repository 활용
- DB 데이터가 AI 모델로 전송되지 않음 (보안)

## 6. 구현 단계

### Phase 1: 기반 구축 (1일)
- [ ] Ollama 설치 및 모델 다운로드
- [ ] Spring Boot에서 Ollama API 연동 서비스 클래스 작성
- [ ] 프롬프트 템플릿 설계 (자연어 → JSON 변환 규칙)

### Phase 2: 백엔드 개발 (1~2일)
- [ ] AiSearchController 생성 (POST /api/ai/search)
- [ ] AiSearchService 생성 (Ollama 호출 + 결과 파싱)
- [ ] UnifiedSearchService 생성 (파싱된 조건으로 기존 Repository 조회)
- [ ] 응답 DTO 설계 (도메인별 통합 결과)

### Phase 3: 프론트엔드 개발 (1일)
- [ ] 통합 검색창 UI (헤더 또는 대시보드)
- [ ] 검색 결과 표시 영역 (도메인별 탭 또는 카드)
- [ ] 로딩 상태 표시
- [ ] 검색 이력 저장 (선택)

### Phase 4: 테스트 및 최적화 (1일)
- [ ] 다양한 자연어 입력 테스트
- [ ] 프롬프트 튜닝 (정확도 향상)
- [ ] 응답 속도 최적화
- [ ] 에러 핸들링 (AI 서버 다운 시 기존 검색으로 폴백)

### 총 예상 기간: 3~5일

## 9. 보안

- 모든 AI 처리는 로컬 서버에서 수행
- 외부 API 호출 없음, 인터넷 불필요
- DB 데이터가 AI 모델로 직접 전달되지 않음 (검색 조건만 생성)
- 기존 Spring Security 인증/인가 그대로 적용

---

## 10. 향후 확장 가능성

1. **대화형 검색**: 검색 결과에 대한 추가 질문 지원
2. **보고서 자동 생성**: 검색 결과를 바탕으로 요약 보고서 작성
3. **이상 탐지**: 만료 예정 라이선스, 지연된 업무 자동 알림
4. **데이터 분석**: "올해 계약 금액 추이" 같은 분석 질의 지원
