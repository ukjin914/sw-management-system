## 1. 배경 / 목표

### 배경
현재 점검내역서(`doc-inspect.html`)의 **5. DBMS 점검결과** 섹션 하단에는 자유 서술용 "기타" 텍스트영역이 존재하여, 점검자가 해당 섹션에 관한 특이사항·추가 메모를 남길 수 있다. 그러나 **7. 표준시스템 점검결과** 섹션에는 동일한 기타 메모 영역이 없어, 표준시스템(UPIS/UPIS_SW/KRAS) 점검 중 발생한 특이사항을 기록할 공간이 없다.

사용자(점검자)는 표준시스템 점검 항목 외의 이슈(예: 특정 메뉴 응답 지연, 화면 깨짐, 운영자 요청 사항 등)를 기록해야 할 필요가 반복적으로 발생했다.

### 목표
- 7번 섹션(표준시스템 점검결과) 하단에 **기타 메모 텍스트영역 1개 추가**.
- 저장·불러오기·PDF 출력·상세보기·미리보기 모든 경로에서 동일하게 표시되어야 한다.
- 5번(DBMS 기타) 와 **시각적·기능적으로 동일한 UX** 제공.

---

## 2. 사용자 시나리오

1. 점검자가 점검내역서 작성 페이지에서 7번 섹션(표준시스템 점검결과) 항목들을 체크한다.
2. 표 아래에 있는 **"기타"** 제목 행 + 텍스트영역을 보고, 특이사항을 자유 서술로 입력한다 (예: "KRAS 연계 건축물대장 조회 시 평균 응답 8초 이상 지연 관찰").
3. 저장 버튼을 누르면 해당 메모가 서버에 저장된다.
4. 이후 **상세보기 / 미리보기 / PDF 다운로드** 화면에서도 해당 메모가 7번 표 아래에 표시된다.
5. 재작성(수정) 진입 시 이전에 저장한 메모가 텍스트영역에 자동 복원된다.

## 3. 화면 / UI 흐름

### 3-1. 편집 화면 (doc-inspect.html)

현재 구조:
```
┌─────────────────────────────┐
│ 5. DBMS 점검결과            │
│   [점검 항목 테이블]        │
│   [기타: textarea]    ← 기존 │
├─────────────────────────────┤
│ 6. GIS 엔진 점검결과        │
│   [점검 항목 테이블]        │
├─────────────────────────────┤
│ 7. 표준시스템 점검결과      │
│   [점검 항목 테이블]        │
│                              │ ← 여기에 [기타: textarea] 추가
└─────────────────────────────┘
```

### 3-2. 스타일 명세
- 5번 DBMS 기타와 **완전히 동일한 마크업** 사용 (일관성 확보).
- `<table class="check-table">` 안에 `<thead>` 로 "기타" 타이틀 셀 (배경 `#e3f2fd`), `<tbody>` 에 `<textarea rows="4">` 1개.
- `placeholder`: "기타 사항 입력"
- `id`: `app_etc` (DBMS 측의 `dbms_etc` 와 대칭)
- `maxlength`: **500** (DB `inspect_check_result.result` VARCHAR(500) 컬럼 길이와 일치)
- 초과 입력 시: 브라우저 기본 차단 + 500 자에 근접(95% 이상) 시 잔여 글자 수 안내 문구 표시

### 3-3. 탭/시스템 유형에 따른 표시 여부
- 7번 섹션 타이틀(`appTitle`)과 테이블(`appCheckTable`)은 **유형별로 표시/비표시**가 전환된다 (기존 로직 유지).
- **기타 영역도 동일한 가시성 규칙**을 따라야 한다 — 즉, 7번 섹션이 숨겨질 때 같이 숨기고, 보일 때 같이 보인다.

### 3-4. 읽기 전용 화면 (상세/미리보기/PDF)
- 7번 표 아래에 **"기타" 제목 행 + 메모 텍스트** 로 출력.
- 메모가 비어 있으면 해당 블록 자체를 숨긴다 (공백 영역 방지).
- **개행·공백 보존 렌더링 필수**: 출력 컨테이너에 `style="white-space: pre-wrap; word-break: break-word;"` 적용. DBMS 기타의 렌더링 규칙과 동일하게 유지한다.
- PDF 템플릿(`pdf-inspect-report.html`) 도 동일한 `pre-wrap` 을 `<div>` 인라인 스타일로 지정한다 (Thymeleaf → PDF 엔진이 CSS 클래스 일부를 누락하는 경우를 방지).

## 6. DB팀 자문 결과

### 영향받는 테이블/컬럼
- **`inspect_check_result`** 테이블 (실제 스키마명, `db_init_phase2.sql:107`)
  - 신규 데이터: `section='APP_ETC'` 인 행이 리포트당 최대 1건 추가
  - 저장 값은 `result VARCHAR(500)` 컬럼에 들어감 — 기획서 FR-10 의 maxlength=500 과 일치
  - 컬럼 변경 **없음** (기존 `section VARCHAR(20)` 재사용, `APP_ETC` 는 7자로 여유 있음)

### 마이그레이션 필요 여부
- **불필요** — 스키마 변경이 전혀 없음.
- 과거 리포트는 `APP_ETC` 행이 없는 상태로 남아 있어도 정상 (읽기 화면에서 빈 값으로 처리).

### 인덱스/제약 변경
- 변경 없음. 기존 인덱스 활용:
  - `idx_check_result_report (report_id)`
  - `idx_check_result_section (report_id, section)` — `APP_ETC` 섹션 조회도 이 인덱스로 커버

### 데이터 정합성 리스크
- 없음. `APP_ETC` 는 새로운 섹션 코드이므로 기존 데이터와 충돌하지 않음.
- 중복 저장 방지: 기존 `DBMS_ETC` 와 동일하게 리포트 저장 시 섹션별 행이 한 번에 치환되므로 중복 리스크 없음.
- 500 자 초과 위험: 프론트 `maxlength=500` 으로 1차 차단, 백엔드에서 추가 검증은 불필요 (DB 레벨 VARCHAR(500) 이 최종 안전망).

**DB팀 의견: 승인 가능 (변경 없음).**

## 7. 의사결정 결과 / 향후 확장

### 7-1. 섹션 코드 네이밍 — ✅ 확정
- **결정**: `APP_ETC`
- 근거: 내부 변수명이 이미 `appTitle`, `appCheckTable`, `appCheckBody` 로 통일되어 있어 일관성 최고.
- 적용 범위: 프론트 JS 저장/로드, 백엔드 필터(`DocumentController`, `InspectPdfService`), 템플릿 모두 이 코드 한 가지만 사용 (FR-12).

### 7-2. 7번 섹션이 화면에 안 보일 때의 기타 영역 처리 — ✅ 확정
- **결정**: 화면 숨김 + 저장 payload 에서도 제외 (FR-7, FR-9).
- 근거: 점검 대상이 아닌 섹션에 메모만 남기는 것은 의미 없고, 숨겨진 값이 DB 에 흘러들어가는 것을 원천 차단.

### 7-3. 기존 DBMS 기타의 숨김 처리 참조 — 개발 단계 확인 항목
- 개발팀이 구현 전 `dbms_etc` 가 DBMS 섹션과 함께 숨김 처리되는 방식을 확인하고, **동일 방식**을 채택한다.

### 7-4. 향후 확장
- 6번(GIS 엔진 점검결과)에도 동일 패턴의 기타가 필요해질 수 있음. 이번 구현이 향후 `GIS_ETC` 를 추가할 때의 템플릿이 되도록 **네이밍 컨벤션과 저장 방식을 일관되게** 유지.
